HTTP Notes

15th January, 2026

A comprehensive overview of the HyperText Transfer Protocol (HTTP), covering its evolution, core concepts, and features.

1. Introduction

HTTP (HyperText Transfer Protocol) is an application-layer protocol for transmitting hypermedia documents, such as HTML. It was designed for communication between web browsers and web servers but is also used for other purposes. It follows a classical client-server model.

2. HTTP Evolution (Key Differences)

  • HTTP/0.9: The original version. One-line protocol. Supported only GET method. No headers. Raw HTML response.
  • HTTP/1.0: Introduced metadata (headers), status codes, and methods (POST, HEAD). Not persistent by default (one connection per request).
  • HTTP/1.1: Standardized persistent connections (keep-alive), pipelining, chunked transfer encoding, and caching mechanisms.
  • HTTP/2: Binary protocol (not text). Multiplexing (parallel requests over one connection), header compression (HPACK), and server push.
  • HTTP/3: Built on QUIC (UDP) instead of TCP. Solves "head-of-line blocking" in TCP. Better performance on unreliable networks.

3. HTTP Messages

HTTP communication consists of requests and responses.

Request Message

GET /index.html HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0
Accept: text/html
Connection: keep-alive

Response Message

HTTP/1.1 200 OK
Date: Mon, 27 Jul 2009 12:28:53 GMT
Server: Apache
Content-Type: text/html
Content-Length: 123

<html>...</html>

4. HTTP Headers

Headers let the client and the server pass additional information with an HTTP request or response.

  • **General Headers:** Apply to both request and response (e.g., `Date`).
  • **Request Headers:** Contain more information about the resource to be fetched, or about the client (e.g., `User-Agent`, `Accept`).
  • **Response Headers:** Hold additional information about the response, like its location or about the server (e.g., `Server`, `Location`).
  • **Entity Headers:** Contain information about the body of the resource (e.g., `Content-Length`, `Content-Type`).

Extensibility: Custom headers can be added. Traditionally prefixed with `X-` (e.g., `X-Custom-Header`), though this is deprecated for standardizing new headers. They allow "remote control" configuration of client/server behavior.

5. HTTP Methods

Defines the desired action to be performed on the identified resource.

  • GET: Retrieve data.
  • POST: Submit an entity to the specified resource (often causes a change in state).
  • PUT: Replace all current representations of the target resource with the upload content.
  • PATCH: Apply partial modifications to a resource.
  • DELETE: Deletes the specified resource.

6. Idempotency

An HTTP method is idempotent if an identical request can be made once or several times in a row with the same effect while leaving the server in the same state.

  • Idempotent: GET, HEAD, PUT, DELETE, OPTIONS, TRACE. (e.g., Deleting an object 10 times results in it being gone, same as deleting it once).
  • Non-Idempotent: POST, PATCH (usually). (e.g., Placing an order 10 times might create 10 orders).

7. Options & CORS

OPTIONS Method: Describes the communication options for the target resource.

CORS (Cross-Origin Resource Sharing): A mechanism that uses additional HTTP headers to tell browsers to give a web application running at one origin, access to selected resources from a different origin.

CORS Request

GET /api/data HTTP/1.1
Origin: https://my-site.com

CORS Response

HTTP/1.1 200 OK
Access-Control-Allow-Origin: https://my-site.com

8. Preflight Request

A CORS preflight request is a CORS request that checks to see if the CORS protocol is understood and currently supported by the server. It uses the `OPTIONS` method.

Criteria (when it occurs):

  • Custom headers are present (e.g., `X-Token`).
  • Method is NOT GET, HEAD, or POST.
  • Content-Type is NOT application/x-www-form-urlencoded, multipart/form-data, or text/plain.

9. HTTP Response Codes

  • 1xx (Informational): 100 Continue.
  • 2xx (Successful): 200 OK, 201 Created, 204 No Content.
  • 3xx (Redirection): 301 Moved Permanently, 302 Found, 304 Not Modified.
  • 4xx (Client Error): 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found.
  • 5xx (Server Error): 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable.

10. HTTP Caching

Storing a copy of a given resource and serving it back when requested. Reduces latency and server load.

  • Cache-Control: max-age=3600 (valid for 1 hour).
  • ETag: "v1.0" (Validator).
  • If client has `ETag`, sends If-None-Match: "v1.0". Server returns 304 Not Modified if match.

11. Content Negotiation

The mechanism that is used for serving different representations of a resource at the same URI, so that the user agent can specify which is best suited for the user.
Client sends: Accept: application/json, Accept-Language: en-US.
Server responds with the appropriate Content-Type and language.

12. HTTP Compression

Compresses the content usually using gzip to save bandwidth.

Request

Accept-Encoding: gzip, deflate, br

Response

Content-Encoding: gzip
(Compressed binary data...)

13. Persistent Connections

Allows using a single TCP connection to send and receive multiple HTTP requests/responses, as opposed to opening a new connection for every single request pair.

  • HTTP/1.0: Default was non-persistent. required Connection: keep-alive.
  • HTTP/1.1: Persistent by default. Reduced handshake latency (3-way handshake) and CPU usage.

14. Multipart Requests

Used to send large files or mix data (e.g., file upload + JSON). The body is divided into parts by a boundary. 

Content-Type: multipart/form-data; boundary=boundary-string

--boundary-string
Content-Disposition: form-data; name="file"; filename="image.png"
Content-Type: image/png

[Binary Data]
--boundary-string--

15. SSL / HTTPS / TLS

HTTPS is HTTP over TLS (Transport Layer Security), formerly SSL.

  • **Encryption:** Encrypts the data exchanged (prevents eavesdropping).
  • **Integrity:** Data cannot be modified/corrupted during transfer.
  • **Authentication:** Proves the server is who it claims to be (via Certificates).
